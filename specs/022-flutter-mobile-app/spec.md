## Context

Sprint 6 closed the API at `/api/v1`. The HTML clickable prototype at `mobile/prototype/SignUpFlow iOS v2.html` validated:

- The screen tree (14 screens, 3 volunteer tabs + 4 admin tabs).
- The visual direction (Block-Mono, see `mobile/prototype/brand-spec.md`).
- Each screen tagged its API source so the prototype doubles as the API contract.

Sprint 7 ports the prototype to a real iOS-first Flutter app that consumes the existing FastAPI backend. Android comes in Sprint 8 (or whenever pulled in — Flutter codebase makes it cheap).

## Why Flutter (not native SwiftUI)

| Lens | SwiftUI | Flutter |
|---|---|---|
| Single iOS app | ✓ best polish | ✓ good polish, less native feel |
| iOS + Android single codebase | ✗ requires Android-side reimplement | ✓ ~95% reuse |
| OpenAPI codegen | dart-dio mature; Swift mature too | ✓ |
| State mgmt ecosystem | Combine/SwiftData | Riverpod 2.x — battle-tested |
| Team scale | needs iOS-specific dev | one engineer ships both platforms |
| Brand spec is custom (not iOS-default) | SwiftUI's native feel goes unused | Material 3 themed = exact match |

Pick: **Flutter**. The brand is custom (Block-Mono, not Cupertino default), so SwiftUI's strongest argument — "looks native" — is moot. Cross-platform reuse is the deciding factor.

## Stack

| Concern | Pick | Rationale |
|---|---|---|
| Framework | Flutter 3.24+ stable | LTS, mature null-safety era |
| Widget kit | **Material 3** with deep custom theme | Custom brand, not Cupertino-default. Material's `Scaffold`, `BottomNavigationBar`, `ListView.builder` theme cleanly to Block-Mono. |
| State | **Riverpod 2.x + riverpod_generator** | `@riverpod`-annotated providers, `AsyncValue` for API loading states, no global singletons |
| Routing | **GoRouter** | Declarative, supports redirects (role-aware), deep links (invitation tokens) |
| Networking | **dio** + retrofit-style API codegen via OpenAPI | Generated from `tests/contract/openapi.snapshot.json` |
| Codegen | `@openapitools/openapi-generator-cli` (npm), generator `dart-dio` | Most mature path for OpenAPI 3 → Dart |
| Fonts | `google_fonts` package | Inter + JetBrains Mono per brand spec |
| Auth storage | `flutter_secure_storage` | iOS Keychain backend |
| Dates/timezones | `intl` + `timezone` | Server stores UTC; client renders local |
| Testing | `flutter_test`, `mocktail` | Standard |

## OpenAPI codegen wiring

**Make target** (already stubbed in `Makefile` per `update-openapi-snapshot` message):

```makefile
mobile-codegen: check-poetry
	@echo "🔄 Generating Flutter API client from OpenAPI snapshot..."
	@npx @openapitools/openapi-generator-cli generate \
	  -i tests/contract/openapi.snapshot.json \
	  -g dart-dio \
	  -o mobile/lib/api \
	  --additional-properties=pubName=signupflow_api,pubVersion=0.0.1,nullSafe=true \
	  --skip-validate-spec
	@cd mobile && flutter pub get && dart run build_runner build --delete-conflicting-outputs
	@echo "✅ Mobile API client regenerated."
```

**Workflow**: when backend ships an API change, the OpenAPI snapshot refreshes (existing `update-openapi-snapshot` target), reviewer runs `make mobile-codegen` to regen the Dart client. Generated code is committed (so the mobile build doesn't fork on codegen drift).

## Auth flow

1. `POST /auth/login` → `{ token, person: { id, roles: [...] } }`.
2. Token stored in `flutter_secure_storage` (iOS Keychain). Role cached in a Riverpod provider.
3. `GoRouter`'s `redirect` reads auth state:
   - Unauthenticated + non-public route → `/login`.
   - Authenticated volunteer → routes under `/v/*`.
   - Authenticated admin → routes under `/a/*`. (Admin can read `/v/*` too; volunteer cannot read `/a/*`.)
4. dio `Interceptor` injects `Authorization: Bearer <token>` on every request; on 401 → wipe token, redirect to `/login`.
5. Token refresh: NOT implemented in v1 (24h JWT expiry → force re-login). Add in Sprint 8 with refresh endpoint.
6. Invitation deep-link: register custom URL scheme `signupflow://invite?token=...` in iOS `Info.plist`. GoRouter handles `/invite` route, calls `GET /invitations/verify`, then `POST /invitations/accept`.

## Out of scope for Sprint 7

- **Android** (Sprint 8 or later — Flutter makes it cheap, but not free; iOS first to focus polish).
- **Push notifications** (no `/notifications` endpoint backend-side; deferred).
- **Offline mode / local cache beyond auth** (online-only v1).
- **Dark mode** (Sprint 8 — needs brand-spec dark variant).
- **Inbox screen** (deferred from MVP, awaits backend endpoint decision).
- **Token refresh** (24h re-login is acceptable for v1).
- **Real-time sync** (no websocket layer; pull-to-refresh + manual reload).
- **In-app schedule editing** (admin uses solver flow only; manual editing is spec 017's concern).
